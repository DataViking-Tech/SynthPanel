# synth-panel 0.4.0 — CPO Vision

**Author:** synthpanel/crew/cpo | **Date:** 2026-04-05

---

## What 0.3.0 Delivered (and What It Changed)

0.3.0 was the synthesis release. It turned synth-panel from "run N prompts in parallel" into "get a research finding." What landed:

| Feature | Impact |
|---------|--------|
| Panel synthesis | Post-panel synthesized finding with themes, agreements, disagreements, surprises, recommendation |
| Conditional follow-ups | `response_contains` / `always` / `never` — first branching primitive |
| Comparative output | Terminal table matrix (personas x questions) with synthesis summary row |

The synthesis was the inflection point. Before it, synth-panel output required human interpretation. After it, the tool produces an opinionated finding. This matters most for the agent surface — an AI agent calling `run_panel` now gets back a decision-relevant summary, not a wall of text to parse.

But 0.3.0 also revealed the product's current ceiling: **every panel run is a single shot.** You define questions, run them, get a synthesis, done. There's no way to say "that was interesting — dig deeper on the pricing objection." The researcher (human or agent) has to manually construct a new instrument, re-run, and mentally connect the findings.

That's the gap 0.4.0 closes.

---

## The 0.4.0 Question

The original roadmap said 0.4.0 was "multi-round conversations: persona responses in round N shape questions in round N+1." The execution plan scoped it as `response_sentiment` + multi-round.

**Is multi-round still the right bet?** Yes — but the framing needs updating now that agent-native is a core identity pillar and synthesis has landed.

The original framing was instrument-centric: "branching logic, multi-round conversations where persona responses in round 1 shape questions in round 2." That's the mechanism. The product framing is different:

**0.4.0 is the release where synth-panel becomes conversational.** Not in the chat sense — in the research sense. A good researcher doesn't just ask questions and leave. They probe. They follow threads. They hear something surprising and dig in. 0.4.0 gives synth-panel that capability.

## 0.4.0 Identity: The Conversation Release

### The Headline: Multi-Round Instruments

A multi-round instrument defines rounds, not just questions. Each round can reference the synthesis from the previous round:

```yaml
instrument:
  version: 2
  rounds:
    - name: exploration
      questions:
        - text: "What frustrates you about project management tools?"
        - text: "How do you currently track work across teams?"

    - name: probe
      depends_on: exploration
      questions:
        - text: "The panel flagged '{synthesis.themes[0]}' — can you describe a specific instance?"
        - text: "Several of you disagreed on '{synthesis.disagreements[0]}'. What's driving your perspective?"

    - name: validation
      depends_on: probe
      questions:
        - text: "If a tool addressed {synthesis.recommendation}, would you switch from your current setup?"
          follow_ups:
            - text: "What would the switching cost look like for you?"
              condition: "response_contains: yes"
            - text: "What would it take to change your mind?"
              condition: "response_contains: no"
```

**Why this matters:** This is the feature that makes synth-panel impossible to replicate by prompting ChatGPT with personas. Anyone can run 5 parallel prompts. Nobody can easily orchestrate a multi-round research conversation where round 2 questions are shaped by round 1 findings, with conditional follow-ups, across multiple personas, with per-round synthesis.

**Why this matters for agents:** An agent calling `run_panel` with a multi-round instrument gets a depth of signal in a single tool call that would otherwise require multiple calls with manual question design between them. The agent doesn't need to orchestrate rounds — synth-panel does it. The tool call becomes: "run this 3-round study and give me the final synthesis." That's a fundamentally more powerful research primitive.

### Three Features, Priority Order

#### 1. Multi-Round Orchestration (the headline)

The orchestrator gains a round loop: run round N, synthesize, template round N+1 questions with synthesis data, repeat.

**Key design decisions:**

- **Synthesis is mandatory between rounds.** Each round produces a synthesis that feeds into the next round's question templates. This isn't optional — it's the mechanism that connects rounds. Within a round, `--no-synthesis` on the final round is still available.
- **Persona continuity across rounds.** 0.2.0 already established single-session-per-panelist (conversation context accumulates). Multi-round extends this — the same persona session carries across rounds. Sarah from round 1 remembers what she said when round 2 asks a follow-up. This is already architecturally possible; the orchestrator just needs to reuse the session.
- **Template syntax for synthesis references.** `{synthesis.themes[0]}`, `{synthesis.recommendation}`, etc. Keep it simple — Python string formatting with the synthesis dict. Don't invent a template language.
- **Round-level synthesis + final synthesis.** Each round gets its own synthesis. The final output includes all round syntheses plus an overall synthesis that looks across the entire conversation.

**What this means for the MCP contract:**

```json
{
  "rounds": [
    {
      "name": "exploration",
      "results": [...],
      "synthesis": { "summary": "...", ... }
    },
    {
      "name": "probe",
      "results": [...],
      "synthesis": { "summary": "...", ... }
    }
  ],
  "final_synthesis": {
    "summary": "Across 3 rounds, the panel converged on...",
    ...
  },
  "total_cost": "$0.12"
}
```

Single-round instruments (the current format) continue to work — they're just a one-round instrument with no template references. No breaking changes.

#### 2. `response_sentiment` Condition (deferred from 0.3.0)

The architect deferred this from 0.3.0 because it requires an LLM call per evaluation. With multi-round already adding LLM calls between rounds, the latency budget is already relaxed — adding sentiment evaluation is now proportionally cheaper.

```yaml
follow_ups:
  - text: "What would make it worth the price?"
    condition: "response_sentiment: negative"
  - text: "What's the most valuable aspect?"
    condition: "response_sentiment: positive"
```

**Implementation:** An async evaluator in the conditions registry that makes a haiku call with a simple "classify this response as positive/negative/neutral" prompt. Cache results per-response so the same response isn't classified twice.

This completes the conditional follow-up system from 0.3.0. With `response_contains`, `response_sentiment`, `always`, and `never`, instruments can express most meaningful branching logic without full conversation trees.

#### 3. Panel Resume / Extend

After a panel run completes, allow extending it with additional rounds:

```bash
# Run initial study
synth-panel panel run --pack startup-founder --instrument study.yaml --save-session my-study

# Review synthesis, then extend with follow-up round
synth-panel panel extend my-study --questions "Tell me more about the pricing concern"
```

**MCP equivalent:**

```json
{
  "tool": "extend_panel",
  "input": {
    "session_id": "result-abc123",
    "questions": ["Tell me more about the pricing concern"]
  }
}
```

**Why this matters:** This is the agent-native killer feature for 0.4.0. An agent can:
1. Run a panel (`run_panel`)
2. Read the synthesis
3. Decide it needs more depth on a specific theme
4. Extend the panel (`extend_panel`) — same personas, same conversation context, new questions

Without this, the agent has to construct a new instrument, re-establish persona context, and lose the conversational thread. With this, the agent can have an iterative research conversation through tool calls. This is what "agent-native" means in practice — the tool's interface is designed for how agents work (iterative, tool-call-driven), not just how humans work (batch, run-once).

## What 0.4.0 is NOT

- **Not full branching instruments / conversation trees.** That's 0.5.0. Multi-round is linear: round 1 → synthesis → round 2 → synthesis → round 3. Branching (if synthesis shows X, go to round A; if Y, go to round B) is a different complexity class.
- **Not persona-to-persona interaction.** Personas still respond independently. They don't debate each other. That's architecturally complex and the CPO doesn't believe the quality of LLM-to-LLM debate justifies the cost and latency.
- **Not auto-generated follow-up questions.** The template system lets instrument designers reference synthesis data, but the questions are still human-authored. Letting the LLM auto-generate follow-ups is tempting but makes results non-reproducible and hard to audit.
- **Not a hosted service or API.** Still CLI + MCP. Still bring-your-own-key.

## Architect Questions

These are product decisions the architect needs clarity on before designing 0.4.0:

1. **Template syntax scope.** Should templates only access `synthesis.*` or also `round[N].results[persona].response`? The broader the syntax, the more powerful but harder to validate. **CPO recommendation:** Start with `synthesis.*` only. It's the 80% case and it's simple. If users need per-persona references, that's 0.5.0.

2. **Round-level cost budgets.** Should the `budget` parameter (existing) apply per-round or to the entire multi-round run? **CPO recommendation:** Entire run. A 3-round study with a $0.50 budget should stop at $0.50 total, not $0.50 per round. But expose `round_cost` in the output so users can see the breakdown.

3. **`extend_panel` persistence.** When a panel is extended, does it create a new result or append to the existing one? **CPO recommendation:** Append. The result ID stays the same. The agent that started the panel can extend it without managing multiple result IDs. The result grows — it's a conversation, not a series of snapshots.

4. **Synthesis model escalation.** In a multi-round instrument, should the final synthesis use a more capable model than intermediate syntheses? **CPO recommendation:** Yes, as a default. Intermediate syntheses (between rounds) use whatever the panelist model is. Final synthesis uses sonnet (or the `--synthesis-model` override). This keeps intermediate costs low while ensuring the final finding is high quality.

## Summary

0.4.0 is the conversation release. Multi-round instruments are the headline — the feature that makes synth-panel impossible to replicate with manual prompting. `response_sentiment` completes the conditional follow-up system. Panel resume/extend makes the agent surface iterative.

Multi-round is still the right bet. What's changed is the framing: it's not just "instrument sophistication." It's the feature that makes the agent-native identity real at the research level, not just the API level. An agent that can run a 3-round study and extend it based on findings is doing something that no other tool enables.

Hold Show HN for 0.4.0 if it hasn't happened yet. The multi-round demo is materially stronger.
