# synth-panel 0.3.0 — CPO Vision

**Author:** synthpanel/crew/cpo | **Date:** 2026-04-04

---

## What 0.2.0 Actually Delivered

Before planning forward, acknowledge what landed — it exceeded the original roadmap:

| Feature | Impact |
|---------|--------|
| `run_prompt` MCP tool | Simplest agent entry point — no personas needed |
| `pack_id` in `run_panel` | One-step panel invocation for agents (eliminated 3-step workflow) |
| Structured output (MCP + CLI) | Responses as validated JSON, not free text |
| REPL wired to runtime | Interactive exploration works end-to-end |
| 5 bundled persona packs | Zero-setup experience — `pip install` includes usable personas |
| Pack validation + CLI commands | `pack list`, `pack import`, `pack export` |
| Single session per panelist | Conversation context accumulates across questions |
| Instrument version field | Schema ready for v2 instrument format |
| Shared prompt builders | Refactored duplicated logic between CLI and MCP |

The original roadmap said 0.2.0 was "REPL wired up, plugin integration tests, first community feedback." What shipped was the agent-native foundation *plus* the persona pack ecosystem *plus* the REPL. 0.2.0 was three releases worth of work.

## The 0.3.0 Question

The CPO-VISION.md roadmap jumps from 0.2.0 to 0.5.0 ("multi-round conversations, branching instruments, session management"). That's too big a leap. 0.3.0 needs to be the stepping stone — the release that makes the Show HN demo undeniable.

The architect identified instrument sophistication as the Phase 2 bet. **That's still right, but the scope needs sharpening.** Full branching instruments and multi-round conversations are 0.5.0 work. The question for 0.3.0 is: what's the smallest instrument upgrade that makes synth-panel feel like a *research tool* rather than a *parallel prompter*?

---

## 0.3.0 Identity: The Synthesis Release

**The insight:** Right now, synth-panel gives you 5 separate persona responses to each question. That's interesting but not actionable. You, the human (or the agent), have to read all of them, spot the patterns, and decide what matters. That's the work synth-panel should be doing.

**0.3.0 is the release where synth-panel stops being "run 5 prompts in parallel" and becomes "get a research finding."**

### Three features, in priority order:

#### 1. Panel Synthesis (the headline feature)

After all panelists respond, synth-panel generates a synthesis: themes, agreements, disagreements, surprise insights, and a recommendation. This runs as a final LLM call using the panelist responses as input.

**Why this matters for Show HN:** The demo goes from "here are 5 AI responses" (meh) to "here's what the panel found, with supporting quotes" (compelling). That's the difference between a toy and a tool.

**Why this matters for agent-native:** When an AI agent calls `run_panel` mid-task, it needs a *decision-relevant summary*, not 5 raw transcripts to parse. The synthesis is what makes synth-panel an opinionated tool call — the agent gets back "3 of 5 personas flagged onboarding friction, here's the pattern" instead of a wall of text it has to interpret. This is the feature that makes the agent integration surface genuinely useful rather than just technically possible.

**Implementation shape:**
- New `synthesis` field in panel run output (alongside individual responses)
- Configurable synthesis prompt (default provided, user-overridable)
- Synthesis model can differ from panelist model (use sonnet for synthesis, haiku for panelists)
- MCP: `run_panel` and `run_quick_poll` return synthesis by default
- CLI: `--no-synthesis` flag to skip it, `--synthesis-model` to override

#### 2. Conditional Follow-ups (the first branching primitive)

v0.1 follow-ups fire unconditionally — every persona gets every follow-up. v0.3 adds a `condition` field:

```yaml
questions:
  - text: "Would you pay for this?"
    follow_ups:
      - text: "What price feels right?"
        condition: "response_contains: yes"
      - text: "What would change your mind?"
        condition: "response_contains: no"
```

This is the smallest meaningful step toward instrument sophistication. It doesn't require multi-round orchestration or conversation branching — it's a filter on the existing follow-up mechanism. But it makes instruments feel *designed* rather than flat.

**Why this matters for Show HN:** Demonstrates that synth-panel does research design, not just prompting. The instrument file becomes a real artifact, not just a question list.

**Implementation shape:**
- `condition` field in follow-up schema (instrument v2)
- Simple condition types: `response_contains`, `response_sentiment`, `always`, `never`
- Evaluation happens in the orchestrator after each main question response
- No breaking changes — omitting `condition` means `always` (backward-compatible)

#### 3. Comparative Output Formats

Right now output is text or JSON. Add a `--compare` mode that renders a matrix: personas as columns, questions as rows, with color-coded sentiment. For the CLI user browsing results in a terminal, this is the "wow" moment.

**Implementation shape:**
- `--output-format compare` or `--compare` flag
- Renders a terminal table (use built-in formatting, no new deps)
- Works with synthesis: summary row at the bottom
- JSON output includes a `comparison` field with the matrix structure

## What 0.3.0 is NOT

- **Not multi-round conversations.** That's 0.4.0 or 0.5.0 — it requires session state management across rounds and fundamentally changes the orchestrator.
- **Not persona interaction/debate.** Personas responding to each other is interesting but premature and architecturally complex.
- **Not auto-generated personas.** Users define their own — that's still a feature, not a limitation.
- **Not a web UI.** The synthesis and comparison features serve the CLI and agent surfaces. No dashboard.

With 0.3.0, the Show HN demo becomes:

That's not "here are some AI responses." That's a research finding. That's Show HN material.

## Agent-Native Implications

The architect should note: synthesis changes the agent integration contract. Today, `run_panel` returns a list of individual responses. With 0.3.0, it returns a structured finding. This means:

- The `run_panel` MCP tool response schema needs a `synthesis` top-level field
- Agents can key decisions off the synthesis without parsing individual responses
- The `run_quick_poll` tool becomes genuinely useful — quick poll + synthesis = actionable signal in one tool call
- Consider: should synthesis be the *default* for MCP and opt-in for CLI, or default for both? **Recommendation: default for both, `--no-synthesis` to skip.** The synthesis is the product.

- **0.1.0** — Go-public. Core works, honest README. **(shipped)**

- **1.0.0** — When someone has built something real on top of synth-panel and the API has been stable for 3+ minor releases.

## Summary

0.3.0 is the synthesis release. It's the release that turns synth-panel from "run N prompts in parallel" into "get a research finding." Panel synthesis is the headline — it makes both the CLI demo and the agent tool call genuinely useful. Conditional follow-ups add the first instrument sophistication. Comparative output makes the terminal experience compelling.

This is the Show HN release. Not 0.2.0 (infrastructure), not 0.4.0 (complexity). 0.3.0 is where the product becomes undeniable: install it, point it at a persona pack, get a synthesized research finding in 30 seconds for $0.03.
