**Author:** synth-panel/architect

## TL;DR

**All three features are feasible. Branching is the right call. No blockers.**

The 0.4.0 foundation lines up well: `instrument.py` already normalizes v1→Round, `templates.py` already has the safe-format vocabulary that branching predicates will reuse, `synthesis.SynthesisResult` exposes `themes: list[str]` / `recommendation: str` / `disagreements: list[str]` natively — exactly the fields the vision's predicates target. Session reuse, conditional follow-ups, and the rounds runner are all wired in `orchestrator.py` (verified: `evaluate_condition()` is now called at lines 349–356; the bug flagged in 040 is fixed).

The branching engine is ~150 lines of new code on top of the existing rounds runner. The instrument library is mostly a clone of the persona-pack code path. Flat-output removal is mechanical.

The architectural risk surface is small and the things to watch are all in *predicate semantics* and *DAG validation messaging*, not in plumbing. CPO recommendations on the six questions are all sound; I'm endorsing all six with minor refinements on #1, #2, and #6.

## Answers to Architect Questions

### 1. Predicate evaluation: hand-roll vs. expression library

**Endorse hand-roll. With one refinement.**

The vision lists three operators (`contains`, `equals`, `matches`) plus `else`. Hand-rolling these is ~40 lines including the parser. Pulling in `simpleeval` or similar costs us:
- A new dependency in a project that currently has two (`httpx`, `pyyaml`)
- A surface area we don't control (operator overloads, attribute access, sandbox-escape CVEs)
- The exact "what does this expression even mean" debugging tax the vision §"Disagreements" already wants to avoid

**Refinement:** Define the predicate grammar as a structured dict, *not* a string. This is the load-bearing call.

```yaml
# GOOD — structured, no parser, no eval
route_when:
  - if: { field: themes, op: contains, value: pricing }
    goto: probe_pricing
  - if: { field: themes, op: contains, value: integrations }
    goto: probe_integrations
  - else: probe_general

# AVOID — string expression, requires a parser
route_when:
  - if: "synthesis.themes contains 'pricing'"
    goto: probe_pricing
```

The vision's example uses the string form (`"synthesis.themes contains 'pricing'"`). I'd push back on that. A structured form has every advantage:

- **No parser.** A string form needs tokenization, quoting rules, and edge-case handling for `'`, `"`, whitespace, escape sequences. A dict form has none of that — YAML did the parsing.
- **Better errors.** "Unknown field `themess`" beats "parse error at column 27."
- **Trivially extensible.** Adding `not_contains` or `gt` is one elif, not a grammar revision.
- **Round-trips through JSON.** The MCP `path` array can echo predicates back to the agent without re-serializing a string.

The string form survives in the docs and the demo as *display syntax* (it reads better in the README), but the on-disk format is structured. The demo predicate `"synthesis.themes contains 'pricing'"` is rendered from `{field: themes, op: contains, value: pricing}` for human consumption.

**Effort:** ~50 lines in a new `synth_panel/routing.py` module: `evaluate_predicate(predicate: dict, context: dict) -> bool` + `route_round(route_when: list, context: dict) -> str`. Three operators, one `else`, one `KeyError` raised on unknown field, tested against the existing `SynthesisResult` shape.

### 2. DAG validation timing

**Endorse parse-time validation. With staged checks.**

Validate at `parse_instrument()`, before any LLM call. The check ladder:

1. **Structural** (already in `instrument.py`): every round has `name` and `questions`, names are unique, `depends_on` points to an earlier round. **Reject.**
2. **Goto resolution** (new): every `goto` target in every `route_when` clause names a defined round. **Reject** with the offending round + target.
3. **Acyclicity** (new): build the directed edge set from `depends_on` and `goto` targets, run a topological sort, fail if any cycle exists. **Reject** with the cycle path printed.
4. **Else completeness** (per Q3): every `route_when` has a final `else` clause. **Reject.**
5. **Reachability** (new): forward-traverse from the entry round and warn (don't reject) on any round that no edge can reach.

**Refinement on the v2-only constraint in `instrument.py:132–136`:** the current parser enforces "depends_on must reference an earlier round" — that's a *linear-multi-round* constraint, not a DAG constraint. v3 instruments need to relax it: a round can depend on any previously-defined round, and `route_when.goto` targets can be defined later in the file (forward references are how routing works at all). This is a parser change, not a behavioral one — the validator handles ordering, not the parser.

**Effort:** ~80 lines in `instrument.py` (v3 branch + DAG validator), reusing the existing `Round` dataclass with two new optional fields: `route_when: list[dict] | None`, and the existing `depends_on` keeps its meaning.

### 3. Missing `else` clause behavior

**Endorse: reject at parse time.**

Silent fall-through is the worst possible default — instruments would skip downstream rounds without anything in the result `path` explaining why. The CPO call is right: every `route_when` block must end in an `else`, validated at parse time, with a clear error message ("round 'probe' has no `else` clause; add one or use `else: <round_name>` to make routing explicit").

A user who genuinely wants "if no clause matches, end the panel here" can write `else: __end__` (or whatever sentinel we pick — see Risks below). Forcing them to spell it out is correct.

### 4. Instrument pack format

**Endorse single-file YAML.**

Persona packs are directories because a pack can have multiple personas + a manifest + (eventually) supporting docs. Instruments are exactly one document. Wrapping a single YAML file in a directory adds friction with no payoff.

**Refinement:** Mirror the persona-pack manifest *fields* inline, even though the format differs. The pack loader code in `mcp/data.py` already understands `{name, version, description, author}` for persona packs. Instrument packs use the same field names at the top level of the YAML, so the loader generalization is "if it's a directory, walk for personas; if it's a file, parse as instrument with manifest keys at top level." That keeps the two pack types semantically aligned for `list_*` / `save_*` / `load_*` MCP tools.

```yaml
# instruments/pricing-discovery.yaml
name: pricing-discovery
version: 1
description: Two-round pricing objection probe with branching on objection type
author: synth-panel/community
instrument:
  version: 3
  rounds:
    - ...
```

Top-level keys are manifest. Nested `instrument:` is the actual v3 instrument.

### 5. Path recording in MCP results

**Endorse: default, not behind a flag.**

The path is the audit trail for a branching run — without it, two runs of the same instrument that took different branches are indistinguishable in the result. That's a reproducibility break. Hiding it behind a flag implies it's optional, which would break exactly the kind of "rerun this exact study" workflow that an agent calling `extend_panel` needs.

The cost is small — `path` is a list of ~5 dicts even for a 4-round branching instrument. The MCP response shape gains one top-level key.

### 6. Final synthesis prompt scope

**Endorse: pass only executed rounds. With one watch-out.**

The CPO's call is correct — the final synthesis describes what happened, not what could have happened. Including skipped branches in the prompt would invite the synthesizer model to hallucinate findings about questions that were never asked.

**Watch-out:** The intermediate-synthesis-vs-final-synthesis distinction from 0.4.0 (intermediate uses panelist model, final uses sonnet) has to extend to the branching case. In a branching run, "final" is the synthesis at the *terminal node of the executed path*, not the synthesis after a fixed last round. The orchestrator needs to track which round is terminal-on-this-path and tag its synthesis as the final one. This is a small bookkeeping change in the rounds runner, but it's the kind of detail that disappears in a vision doc and bites at integration time.

```

├── instrument.py          # +v3 parser, +DAG validator (~80 LOC delta)
├── routing.py             # NEW: evaluate_predicate, route_round (~50 LOC)
├── orchestrator.py        # Rounds runner gains a router call (~30 LOC delta)
├── packs/instruments/     # NEW: bundled instrument packs (5 YAMLs)
├── mcp/
│   ├── data.py            # Generalized pack loader: dir vs. file (~40 LOC delta)
│   └── server.py          # +list/save/load_instrument_pack tools (~80 LOC delta)
├── cli/commands.py        # +`instruments list/install/show/graph` subcommands (~120 LOC delta)
└── output.py              # Drop flat-output branch; --legacy-output emits warning (~40 LOC delta)
```

Total new+changed: ~440 LOC across 7 files. Comparable in scope to 0.4.0's multi-round work.

### Sequencing

The vision implies all three features ship together; that's right, but the *internal* order matters:

1. **Routing engine first** (`routing.py` + tests). Pure, no LLM, fully unit-testable. Lock the predicate semantics before anything depends on them.
2. **v3 parser + DAG validator** (`instrument.py`). Also pure, also fully unit-testable. The "reject bad instruments in milliseconds" property is a test target.
3. **Orchestrator wiring** (`orchestrator.py`). Replaces the linear "for round in rounds" loop with "while next_round: run(next_round); next_round = router.next(...)".
4. **Flat-output removal** (`output.py`). Independent of everything above; can land in parallel. Gated on `--legacy-output` for one release.
5. **Instrument pack loader generalization** (`mcp/data.py`). Independent of branching; the first 5 bundled packs can use linear v2 instruments and be upgraded to v3 once branching lands.
6. **CLI + MCP surface** (`cli/commands.py`, `mcp/server.py`). Depends on 1–5.
7. **Bundled instrument packs** (`packs/instruments/*.yaml`). Authoring + curation work, parallelizable with 1–6, but the packs *that demonstrate branching* depend on 1–4.

The dependency graph means 4 and 5 can run as polecats in parallel with the critical path 1→2→3→6.

## Risks & Trade-offs

### R1 — Predicate string vs. dict bikeshed (LOW, but important to settle)

If we ship the string form the vision shows, we are committing to a parser and a grammar. If we ship the dict form, we're committing to slightly uglier YAML in exchange for zero parser surface. **Recommendation:** dict on disk, string in docs/README rendered from dict. Settle this before any code lands. **Mitigation:** Architect call — go dict.

### R2 — Forward references break parse ordering (LOW)

The current parser enforces `depends_on` to refer to *earlier* rounds. Branching needs forward references in `goto`. **Mitigation:** Two-pass parse — first pass collects all round names, second pass validates references. Trivial change.

### R3 — Predicate against unstructured `themes: list[str]` is fuzzy (MEDIUM)

`synthesis.themes` is a free-text list produced by an LLM. `themes contains 'pricing'` does substring matching against entries. That works for `["pricing concerns", "integration gaps"]` and breaks for `["cost is the dealbreaker", "needs better APIs"]`. The router will route to `probe_general` and the user will think the predicate is broken when really the *upstream synthesizer* didn't normalize.

This is a real risk and the vision doesn't address it. **Mitigation options:**

- **(a)** Document loudly that predicates match against the synthesizer's exact output and recommend that branching instruments include "list themes using these exact tags when applicable: pricing, integrations, ux, performance" in the synthesis prompt. Boring but works.
- **(b)** Add a `theme_taxonomy:` instrument-level field that gets injected into the synthesis prompt as a constrained vocabulary. Fancier; defer to 0.6.0.
- **(c)** Allow the matches operator over a regex on the *full synthesis summary* as an escape hatch when `themes` doesn't carry the signal.

**Recommendation:** Ship (a) as docs in 0.5.0; revisit (b) if user feedback says branching is unreliable. (c) is already in scope via the `matches` operator — make sure it can target `summary` and `recommendation`, not just `themes`.

### R4 — Reachability warnings vs. errors (LOW)

The CPO call says unreachable rounds are a warning, not an error, because authors-in-progress may have dead branches. Right call. **Watch-out:** the warning has to land somewhere visible. CLI: stderr. MCP: a `warnings: []` field on the run result, *not* a log line nobody reads.

### R5 — Path serialization in `--output-format text` (LOW)

The `path` field is JSON-natural but ugly in text output. **Mitigation:** In text mode, render path as `exploration → probe[themes contains 'pricing'] → probe_pricing → validation`. ~20 LOC in `output.py`. Worth doing — the path is the most interesting line in a branching run's output.

### R6 — Flat output removal blast radius (LOW–MEDIUM)

The vision says agents post-date deprecation, so MCP gets the new shape unconditionally. True. But: **the bundled persona packs, examples, and skill definitions** were authored against flat output. **Mitigation:** Audit `examples/`, `skills/`, and any `0.3.0`/`0.4.0` doc references for flat-shape assumptions before removing the branch. Probably 1–2 hours of grep + fix.

### R7 — `else: __end__` sentinel design (LOW)

If we allow "end the run after this round" as a routing target, we need a sentinel. Options: `null`, `__end__`, `END`, omitting `goto`. **Recommendation:** Reserve the round name `__end__` and treat any `goto: __end__` (or `else: __end__`) as "this is a terminal path, run the final synthesis now." Cleaner than null because it shows up in the `path` output as an explicit terminator.

### R8 — Cost bound on a misauthored DAG (LOW; already addressed)

A DAG with no cycles is bounded above by the total round count, which is bounded by the YAML file size. The CPO's "each round runs at most once" rule plus parse-time cycle detection means there is no runtime path explosion. Existing `BudgetGate` (cost.py:305-337) is the second line of defense.

## What I'd Add to the CPO Doc

Three small additions, none of which need a doc revision — flagging here for when the PM writes the execution plan:

1. **`synth-panel instruments graph <file>` is mentioned in passing.** It should be a hard scope item, not a "if anything." A branching instrument is unreadable without it. ~30 LOC using a Mermaid string template, no graphviz dependency. Ship at 0.5.0.

2. **The v3→v2→v1 backward-compat ladder needs a test matrix.** v1 instruments are still in `examples/`, v2 instruments are still in `tests/`, v3 is new. The acceptance tests need to cover all three formats running through the same orchestrator path. The existing `instrument.py` test file is the right place.

3. **`extend_panel` interaction with branching.** If a user calls `extend_panel` on a result that took the `probe_pricing` branch, what does "extend" mean? Does the agent get to specify a round name to add, or does extend always run a fresh ad-hoc round? **My read:** extend is unchanged — it's a single new round appended to the result, not a re-entry into the DAG. The DAG was authored, the extend is improvised. Document this explicitly so the next CPO/agent boundary discussion doesn't relitigate it.

---

## Verdict

**Ship 0.5.0 as scoped.** Branching is the right step change, the instrument library is the right ecosystem play, and the flat-output removal pays down debt at the right release. The hand-roll predicate engine, dict-form predicates, parse-time DAG validation, and "executed-rounds-only" final synthesis are all the boring, defensible calls.

The execution risk is concentrated in R3 (fuzzy theme matching against LLM-generated lists) — and that risk is a *user-experience* risk, not an architectural one. The architecture supports it; the docs need to set expectations. Worth a paragraph in the 0.5.0 README, not a redesign.

No blockers. Estimate parity with 0.4.0 — ~30–40h of focused work, parallelizable across 2–3 polecats once routing.py + the v3 parser land.

Recommend the PM proceed to execution planning.
